# Sockbowl RBAC / Auth Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Enable authentication across the whole stack and enforce a permission-based RBAC model that lives in Keycloak, is provisioned with sensible defaults, and is re-composable without code changes.

**Architecture:** Keycloak realm holds fine-grained permission-roles bundled into composite roles. Both Spring backends validate the same realm-issued JWT (by issuer, no audience check) and enforce with `@PreAuthorize("hasAuthority('<perm>')")`, authorities mapped from `realm_access.roles`. The game backend uses a confidential client-credentials service account to authenticate its server-to-server packet fetch. The Angular UI reflects permissions in routes/controls. Auth defaults ON.

**Tech Stack:** Keycloak 26, Spring Boot 4.1 (Security + OAuth2 resource server), Spring GraphQL, Angular 21 (angular-oauth2-oidc), Docker Compose, bash + Keycloak Admin REST API.

## Global Constraints

- Spring Boot 4.1.0 / Java 17 / Gradle (both backends).
- Realm name: `sockbowl`. Issuer via `KEYCLOAK_ISSUER_URI`. Roles live in `realm_access.roles`.
- JWT validation is issuer-only (NO audience check) so one `sockbowl-game` token works for both backends — mirror game's existing `JwtDecoders.fromIssuerLocation` + `keycloakJwtAuthenticationConverter`.
- Auth is gated by `sockbowl.auth.enabled` (Spring) / `AUTH_ENABLED` (compose); a `SecurityConfig` (on) + `NoSecurityConfig` (off, `matchIfMissing=true`) pair per backend.
- Permission-roles (exact names): `packet:read`, `packet:create`, `packet:update`, `packet:delete`, `question:generate`, `taxonomy:manage`, `game:host`, `user:ban`, `admin:access`.
- Composite roles: `player`={packet:read, game:host}; `author`=player+{packet:create, packet:update, question:generate, taxonomy:manage}; `moderator`=player+{user:ban}; `admin`=author+moderator+{packet:delete, admin:access}. Legacy `user` = alias composite → player. Realm default role → `player`.
- Confidential service client: `sockbowl-game-backend`, service-accounts + `client_credentials`, service-account granted `packet:read`.
- Auth uses bearer `Authorization` header (no cookies → no CORS credentials mode).
- Verification is compile + `./gradlew test` + `docker compose config -q` + live curl against a booted stack. Commit after each task.

## Task 1: RBAC model file + idempotent loader script

**Files:**
- Create: `sockbowl-docker/keycloak/rbac-model.json`
- Create: `sockbowl-docker/scripts/load-rbac.sh`
- Test (manual): boot Keycloak, run loader, assert roles exist via Admin API.

**Interfaces:**
- Produces: realm roles (permission + composite) and the `sockbowl-game-backend` client with service-account `packet:read`, in realm `sockbowl`. Consumed by all backends (they check these authorities) and Task 7 (compose init).

- [ ] **Step 1: Write the RBAC model**

`keycloak/rbac-model.json`:
```json
{
  "permissionRoles": [
    "packet:read", "packet:create", "packet:update", "packet:delete",
    "question:generate", "taxonomy:manage", "game:host", "user:ban", "admin:access"
  ],
  "compositeRoles": {
    "player":    ["packet:read", "game:host"],
    "author":    ["player", "packet:create", "packet:update", "question:generate", "taxonomy:manage"],
    "moderator": ["player", "user:ban"],
    "admin":     ["author", "moderator", "packet:delete", "admin:access"],
    "user":      ["player"]
  },
  "defaultRole": "player",
  "serviceClient": { "clientId": "sockbowl-game-backend", "roles": ["packet:read"] }
}
```

- [ ] **Step 2: Write the loader script**

`scripts/load-rbac.sh` — idempotent upsert via Admin REST API. Reads env: `KEYCLOAK_URL` (default `http://localhost:8080`), `KEYCLOAK_REALM` (`sockbowl`), `KEYCLOAK_ADMIN`, `KEYCLOAK_ADMIN_PASSWORD`, `RBAC_MODEL` (path). Logic:
1. Get admin token from `master` realm (`grant_type=password`, client `admin-cli`).
2. For each `permissionRoles` + each key in `compositeRoles`: `POST /admin/realms/$R/roles` (ignore 409).
3. For each composite: resolve child role reps (`GET /admin/realms/$R/roles/$child`), `POST /admin/realms/$R/roles/$name/composites` with the array (idempotent).
4. Set realm default role: add `defaultRole` as a composite child of `default-roles-$R` (`POST .../roles-by-id/{defaultRolesId}/composites`).
5. Ensure service client: `GET /clients?clientId=sockbowl-game-backend`; if absent `POST /clients` `{clientId, serviceAccountsEnabled:true, standardFlowEnabled:false, publicClient:false, secret from env SOCKBOWL_GAME_BACKEND_SECRET}`. Then find its service-account user (`GET /clients/{id}/service-account-user`) and assign realm role `packet:read` (`POST /users/{saId}/role-mappings/realm`).

Use `set -euo pipefail`, `curl -sf`, `jq`. Guard each create with a GET-first or accept 409. Print a summary line per created/updated role.

- [ ] **Step 3: Verify against a running Keycloak**

Run:
```bash
cd sockbowl-docker
docker compose --env-file .env.example up -d postgres keycloak-realm-init keycloak
# wait for keycloak health, then:
KEYCLOAK_ADMIN=admin KEYCLOAK_ADMIN_PASSWORD=admin123 bash scripts/load-rbac.sh
```
Expected: exits 0; `curl` admin API lists all 9 permission roles + `player/author/moderator/admin/user`; `sockbowl-game-backend` client present with `packet:read` on its service account.

- [ ] **Step 4: Commit**

```bash

git commit -m "feat(rbac): declarative RBAC model + idempotent Keycloak loader"
```

- [ ] **Step 3: Write SecurityConfig (auth on)**

Create `SecurityConfig.java` — copy game's converter verbatim (realm_access.roles → ROLE_* AND raw authorities). NOTE: to support `hasAuthority('packet:read')` where the role name IS `packet:read`, add the raw role name as an authority (not just `ROLE_`-prefixed). Converter adds BOTH `new SimpleGrantedAuthority(role)` and `ROLE_`+role for each realm role. Class:
```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity
@ConditionalOnProperty(name = "sockbowl.auth.enabled", havingValue = "true")
public class SecurityConfig {
  @Bean SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http.csrf(c -> c.disable())
        .cors(Customizer.withDefaults())
        .authorizeHttpRequests(a -> a.anyRequest().authenticated())
        .oauth2ResourceServer(o -> o.jwt(j -> j.jwtAuthenticationConverter(conv())))
        .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS));
    return http.build();
  }
  @Bean JwtAuthenticationConverter conv() { /* realm_access.roles -> role + ROLE_role */ }
}
```

- [ ] **Step 4: Write NoSecurityConfig (auth off)**
